## Outcome

PRD **Journey 1 (Petr)** is now deliverable: a first-time GitLab CI user can run `semvertag doctor` before their first real `semvertag` invocation and get a named, actionable list of what's missing — token, scopes, project access, protected-tag rules — with config-source provenance and a single-line `--json` envelope for CI dashboards. The four sequential checks short-circuit on first failure (subsequent checks marked `skipped`), the FR37-stable exit-code surface holds via `_DOMINANCE = 3 > 4 > 2 > 1` resolution, and every output path (Rich human form, JSON envelope, stderr error) enforces token redaction at construction time — not at render time.

Both stories landed `done` on 2026-05-29. The 2-story split (orchestration → CLI wrapper) was the cleanest epic structure in the project's history.

## Recurring themes

### 1. The pure-Python-then-thin-CLI split was the cleanest epic structure in the project so far
Story 3.1 shipped 36 stmts + 14 branches of pure orchestration logic under a 100% branch gate, with no CLI surface, no DI plumbing, no integration tests. Story 3.2 added 62 stmts + 12 branches of renderer + a Typer subcommand that consumed those pure functions, with 35 new tests across unit + integration layers. The split paid back in four distinct ways:

- 3.1's 100% branch gate was achievable in a single sitting (33 unit tests, no `MockTransport`, no `CliRunner`)
- 3.2's integration tests inherited 3.1's gate as a regression canary (`run_checks` / `resolve_exit_code` behavior pinned)
- Neither story added an `ioc.py` Group — DI complexity stayed at zero
- The orchestration↔CLI boundary had no seam to debug

This contrasts with Story 1.5 (17 patches, 27 review findings in one story) and Story 2.1 (5 decisions + 8 patches across architecture-layer concerns). The 3.x split avoided cross-cutting review pressure by isolating concerns. Pattern is reusable for future epics that split "data shape + pure logic" from "wired entrypoint."

### 2. The 100%-branch gate reshaped 3.1's production code (again)
Initial `resolve_exit_code` ended with `return SemvertagError.exit_code` as a defensive fallback after a `for code in _DOMINANCE: if code in failed_codes: return code` loop. Since `_DOMINANCE` enumerates every exit code that `_exit_code_for_failed_check` can return, the fallback was unreachable — the coverage tool flagged the dead branch and refused to credit 100%. Refactored to `return next(code for code in _DOMINANCE if code in failed_codes)` — concise, no dead branch, but the generator's `StopIteration` would surface as a crash if a future `SemvertagError` subclass with a new exit code is added without extending `_DOMINANCE`.

This is the third time the strict-gate discipline shaped production code: 1.4's `pragma: no cover` on a defensive RuntimeError, 1.6's `Field(min_length=1)` validator addition, 2.1's `_VALID_TYPE_RE` pydantic validator. Pattern is now: **the gate shapes the code, and the trade-off must be flagged as deferred-work for the next contributor.** Story 3.1 did this correctly (`_bmad/deferred-work.md §3-1` item 1 spells out the `StopIteration` risk).

### 3. Story 3.2's review caught a near-leak — defense-in-depth worked
Story 3.2 has token redaction at **five layers**:
1. `pydantic.SecretStr` containment in `Settings` (Story 1.2)
2. `_redact_token` deterministic last-4 inside `build_doctor_result`, baked into `ConfigSourceView.value` (Story 3.2)
3. `_redact.redact(...)` regex sweep on cause cells in `render_doctor_human` (Story 3.2)
4. `Output.error` redaction on stderr error paths (Story 1.3)
5. **`_redact.redact(...)` applied to `str(value)` non-token fields in the configuration table** — added by review patch P2

Layer 5 was the live save. Without it, a developer pasting `SEMVERTAG_GITLAB__ENDPOINT=https://gitlab.example.com/?private_token=glpat-XXXXXXXXXXXXXXXXXXXX` into their shell would have leaked the token into the doctor's configuration row for `gitlab.endpoint` — the value is a plain `str`, not a `SecretStr`, so `_format_setting_value` returned it verbatim. The unit tests passed; the integration tests passed (the test fixture uses a clean URL). Review's Edge Case Hunter caught it by considering pathological env-var shapes.

NFR10 ("tokens never appear in stdout, stderr, log files, or doctor output") survives even pathological inputs. Review continues to be the security backstop — not test coverage. Same pattern as 1.5's SSRF guard on the `Link: rel="next"` follow.

### 4. `schema_version="1.0"` is a new stability surface — locked before public release
The doctor JSON envelope is the **first schema-versioned envelope** semvertag has shipped. The main verb's JSON output (Story 1.3 `RunResult` + Story 1.7 wired entrypoint) predates the schema-version discipline. Story 3.2 introduced `DoctorResult(schema_version: str = "1.0", configuration, checks)` as a frozen kw-only dataclass; tests pin field order via `dataclasses.fields(DoctorResult)` and the first-key invariant via `next(iter(asdict(...).keys())) == "schema_version"`.

Locking to `"1.0"` at v1.0 means any future breaking change to `DoctorResult` field order, field names, or `ConfigSourceView` shape requires `schema_version` bump + deprecation per NFR25. A future contributor who renames `configuration → settings` or reorders `(value, layer, detail) → (layer, detail, value)` will hit loud test failures. Good guard, but the deprecation cycle isn't yet documented externally — Story 4.4 (mkdocs site) and Story 4.6 (trust surface) both need to publish the schema and the change-policy.

### 5. The `Output` protocol stayed at three methods
Architecture line 531 prescribed: *"doctor uses the same `Output` protocol … with one extension — a configuration-section renderer reading `Settings._provenance`."* Story 3.2 implemented the "extension" as a **free function** (`render_doctor_human(result, output)`) consuming an existing `RichOutput.info_console`, NOT as a new Protocol method (`Output.emit_doctor(result)`).

Why this matters: if `emit_doctor` had been added to the Protocol, `JsonOutput` would have needed to grow a method it doesn't use (the JSON form has its own free-function path, `render_doctor_json`), and the main verb's clean three-method contract would have been polluted. The architecture's "informal extension" phrasing was load-bearing — the spec intentionally avoided a protocol grow.

Pattern: when the architecture hints at a protocol extension, prefer free-function + existing protocol surface unless multiple consumers genuinely need polymorphism. The doctor command has one Rich consumer and one JSON consumer; no polymorphism required.

### 6. Cause-fragment ownership held under real-provider integration
Story 3.1 deferred-work item 4 flagged that `_AUTH_CAUSE_FRAGMENTS` / `_CONFIG_CAUSE_FRAGMENTS` / `_PROVIDER_API_CAUSE_FRAGMENTS` substring + case-sensitive matching could collide ("Token missing" mid-prose; lowercase "token rejected" falls through to generic). Story 3.2 integration tests against `MockTransport` exercised each fragment family with real `GitLabProvider` cause output:

- `Token rejected` (AC6: 401 on `/api/v4/user`)
- `Token missing 'api' scope` (AC7: scopes payload missing `api`)
- `GitLab project not found` (AC12 #6: 404 on `/api/v4/projects/{id}`)
- `Token cannot read` (AC12 #7: 401 on `/protected_tags`)
- `Unexpected GitLab response` (500 status retry-exhausted)

No cross-family causes observed. The hypothetical multi-family routing concern (auth → config → provider_api lexical order ignoring `_DOMINANCE`) remains hypothetical for the current GitLab provider — each `check_*` method emits one cause string, single-family. Story 3.2 didn't change cause vocabulary, so 3.1's substring discipline is internally consistent. Coupling-by-convention survives v1.0.

### 7. Pre-existing lint regression discovered mid-implementation — meta-process gap
Story 3.1's post-review 1-line ownership comment in `_checks.py` was 145 chars long. Story 3.1's Dev Agent Record claimed `just lint-ci` clean, but the post-review patch landed **without re-running** the full `lint-ci` + `test` + `uv build` gate — only the test layer was re-run. Story 3.2 caught the E501 violation in Task 5.2 (`just lint-ci` failed) and applied a 1-line comment-wrap fix.

The gap is process: **post-review patches must re-run the full gate, not just targeted tests.** Without the fix, Story 3.2 could not satisfy AC14 (`just lint-ci` must pass), so the work was forced. The 1-line wrap is in `doctor/_checks.py:16` and preserves the cause-fragment ownership comment verbatim across the wrap. Logged as B14 below (process action).

### 8. `_doctor_command` ↔ `_main_callback` byte-identical exception translation = intentional duplication
AC10 of Story 3.2 mandated **byte-identical** exception translation between the two Typer entrypoints. The five except clauses (`pydantic.ValidationError`, `ImportError`, `SemvertagError`, `BrokenPipeError`, `OSError(EPIPE)`) and their `output.error(...)` + `raise typer.Exit(code=err.exit_code) from err` shape are mirrored ~30 lines verbatim in both functions. Story 1.7's design choice that doctor inherits.

Trade-off: single-owner is intentionally violated by syntactic necessity. A future fix to one entrypoint must be applied to the other in lockstep. The current cost is manageable (two sites, ~30 lines each, identical shape). The future cost grows if Story 4.3a (GitHub Actions Marketplace wrapper) or 4.3b (GitLab CI Catalog component) introduce additional Typer command bodies — three+ sites of duplication start to bite. Refactor candidate when the third command lands: `@contextlib.contextmanager translate_semvertag_errors_to_exit_codes(...)` consumed by all entrypoints. (B11 below.)

## Struggles / friction points

1. **Story 3.2 had the heaviest review cycle since Story 1.5** (23 findings: 4 decisions + 7 patches + 6 deferred + 6 dismissed). The matrix of (chain outcome) × (output form) × (overlay state) × (redaction path) is genuinely combinatorial, and the implementation surface was larger than 3.1 (~80-100 LOC + integration tests). Review continues to be load-bearing for security-critical surfaces — and the P2 near-leak shows that load-bearing review prevents real defects, not just polish.
2. **`_redact_token` short-secret disclosure (D3 decision).** A 4-char secret produces `"***" + secret[-4:]` = the whole secret. AC2 wording technically permitted this; review flagged it. Decision converted to: tighten guard to `len <= _TOKEN_SUFFIX_LEN → "***"` instead of `len < _TOKEN_SUFFIX_LEN`. Second time after 2.1's D4 that a defense-in-depth gap was flagged in review — review continues to be the security backstop, not implementation paranoia.
3. **`_doctor_command` ↔ `_main_callback` duplication is fault-pattern coupling.** Five except clauses mirrored byte-identical (~30 lines). Any future fix to error translation requires touching both sites. Drift risk grows if Story 4.3a/4.3b introduce additional Typer commands. Refactor candidate flagged for when the third entrypoint lands. (B11.)
4. **Pre-existing E501 regression from Story 3.1 post-review.** Story 3.1's Dev Agent Record claimed `lint-ci` clean but the post-review 1-line comment was 145 chars. Story 3.2 had to fix it as a Task 5.2 dependency. Gap: **post-review patches don't re-run the full gate** — only the targeted tests. Process issue, not code issue. (B14.)
5. **Cause-fragment ownership is coupling-by-convention.** `_checks.py` fragments shadow `providers/gitlab.py` cause vocabulary. Story 3.2 integration tests pin the current vocabulary against `MockTransport` responses, so drift surfaces in integration test failures — but only if a `gitlab.py` change ships first. If `_checks.py` is changed without a `gitlab.py` change, all tests stay green because integration tests run against `MockTransport`, not real GitLab. The actual production coupling is only verified by hand under real GitLab. Acceptable for v1.0 since `providers/gitlab.py` is single-owner.
6. **`resolve_exit_code` `next()` generator has implicit `StopIteration` exposure.** Trade-off for 100% branch coverage. A future `SemvertagError` subclass with a new exit code added without extending `_DOMINANCE` will crash the resolver. Documented as deferred-work; a 1-line defensive guard (B12) closes it.

## Forward-looking concerns for Epic 4

Epic 4 = Public-Launch Readiness, 8 stories (4.1–4.8). 4.1 (CI workflow polish) has already landed `done` 2026-05-29. Remaining: 4.2 (Trusted Publishing), 4.3a/3.b (Marketplace + Catalog wrappers), 4.4 (mkdocs site), 4.5 (migration guides), 4.6 (trust-surface markdown), 4.7 (README hero + issue templates), 4.8 (shadow-mode parity validation).

**C1 — Story 4.2 introduces a new token surface that doctor doesn't yet check.**
PyPI Trusted Publishing uses an OIDC token in the publish workflow. Doctor v1.0 only checks GitLab tokens. If a future doctor variant wants to validate publish-time tokens (`semvertag doctor --check pypi-publishing`?), it would require a new `Provider.check_*` method and corresponding `_checks.py` / `_render.py` wiring. **Out of v1.0 scope**; flag as v1.x consideration.

**C2 — Story 4.4 mkdocs site must publish the doctor JSON schema.**
`schema_version="1.0"` is locked but undocumented externally. Story 4.4 must publish: field order (`schema_version` → `configuration` → `checks`), redaction rules (deterministic last-4 for `*.token` SecretStr fields; `***` for short/empty), cause-string vocabulary (with stability caveat — "subject to change with `gitlab.py` rewordings; check integration tests for current set"), and deprecation policy (NFR25 follows). (B7.)

**C3 — Story 4.6 trust-surface markdown must enumerate the 5 redaction layers.**
External readers will scan for "how do you guarantee no token leaks." A concrete answer naming each layer with file:line citations (`semvertag/_settings.py` SecretStr, `semvertag/_redact.py:6` regex, `semvertag/doctor/_render.py:_redact_token` last-4, `semvertag/_output.py` error-path redaction, `semvertag/doctor/_render.py` `_format_setting_value` non-token regex sweep) will be more credible than "we have tests for it." (B8.)

**C4 — Story 4.8 shadow-mode must decide whether to auto-run doctor.**
Shadow-mode (per 4.8 scoping) resolves both strategies simultaneously via the `current_strategy` Factory pattern. The doctor command should NOT run when invoked under shadow-mode unless explicitly enabled — doctor's four HTTP round-trips would double the request count to the provider for every shadow invocation, blowing NFR4's 10s budget. Default: shadow-mode skips doctor; explicit flag opts in. (B9 — decision required at 4.8 spec time.)

**C5 — `_doctor_command` ↔ `_main_callback` duplication will start to bite if Story 4.3a/4.3b introduce CLI surfaces.**
Story 4.3a (GitHub Actions Marketplace wrapper) and 4.3b (GitLab CI Catalog component) may expose CLI entrypoints that need similar error translation. If they reuse `MAIN_APP` directly, no new duplication. If they ship separate Typer apps (for SCM-specific wrapper UX), the exception-translation block needs to be a shared utility. Refactor opportunity (B11).
