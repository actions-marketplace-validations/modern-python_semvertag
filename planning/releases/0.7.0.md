# semvertag 0.7.0 — opt-in patch bump for non-merge commits

**Minor release adding an opt-in `branch-prefix` flag that patch-bumps plain (non-merge) commits, plus an internal `httpware` upgrade to 0.12.0.** Both changes are backward-compatible: the new flag defaults off, and the dependency bump changes no observable CLI behavior. Existing users upgrade with no action required.

If you use the `conventional-commits` strategy, or you only want bumps on merge commits, nothing changes for you — skip straight to the upgrade note.

## What landed

- **`patch_on_non_merge_commit` flag on the `branch-prefix` strategy** (default `false`). When enabled, a plain commit pushed directly to the default branch — one whose subject carries no merge mark — bumps **patch** instead of producing no bump. A merge commit with an unrecognized branch prefix still produces no bump; the flag governs only the "not a merge at all" case. Settable via `SEMVERTAG_BRANCH_PREFIX__PATCH_ON_NON_MERGE_COMMIT=true`. ([PR #24](https://github.com/modern-python/semvertag/pull/24))
- **`httpware` floor raised 0.8.2 → 0.12.0** and both Link-header pagination call sites collapsed from `send_with_response(build_request("GET", ...))` to the 0.12.0 `get_with_response(...)` shortcut. Identical requests and return shapes — no behavior change. The version move also picks up 0.11.0's security/correctness hardening (URL secret redaction in resilience-event logs, a RetryBudget give-up fix). ([PR #24](https://github.com/modern-python/semvertag/pull/24))

## Usage

For a team that allows direct pushes to the default branch and wants each push auto-tagged as a patch release:

```sh
export SEMVERTAG_STRATEGY=branch-prefix
export SEMVERTAG_BRANCH_PREFIX__PATCH_ON_NON_MERGE_COMMIT=true
semvertag tag
```

With the flag set, a direct (non-merge) commit on the default branch produces a patch bump; `feature/` merges still bump minor and `bugfix/`/`hotfix/` merges still bump patch, exactly as before. Leave the variable unset (or `false`) to keep the prior merge-only behavior.

## Why

`branch-prefix` infers the bump from a merge commit's branch name, so a commit pushed straight to the default branch — not via an MR/PR — carried no signal and produced no version movement. For teams that permit direct pushes, real changes were landing untagged. The smallest sensible default for "code changed but no merge told me how much" is a patch, and gating it behind an opt-in flag keeps merge-only workflows untouched.

## Breaking changes

None. `patch_on_non_merge_commit` defaults to `false`, so every existing invocation behaves identically to 0.6.x. The `httpware` upgrade is internal — `semvertag`'s CLI, outputs, and exit codes are unchanged.

## Release procedure

`tag-major.yml` automatically updates the floating `v0` tag on release-published.

```sh
gh release create 0.7.0 \
  --title '0.7.0 — opt-in patch bump for non-merge commits' \
  --notes-file planning/releases/0.7.0.md
```

`publish.yml` fires on release creation and pushes 0.7.0 to PyPI (`just publish` runs `uv version 0.7.0`, builds, and uploads). `tag-major.yml` fires in parallel and force-updates `v0` to point at the 0.7.0 commit.
