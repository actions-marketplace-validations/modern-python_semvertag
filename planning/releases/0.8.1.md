# semvertag 0.8.1 — tag-driven release pipeline (no package behavior change)

**Patch release. Infrastructure only — the package is functionally identical to 0.8.0.** The CLI, the `--json` envelope (`schema_version` `1.0`), and the `@v0` GitHub Action outputs (`tag` / `bump` / `status`) are byte-for-byte unchanged. If you consume `semvertag` in any way, this is a transparent upgrade — there is nothing to change.

What changed is *how releases are cut*, not what the tool does. This is the first release published by the new pipeline, and cutting it is the validation that the pipeline works.

## What landed

- **Tag-driven release pipeline** (`.github/workflows/release.yml`). Pushing a bare semver tag now publishes to PyPI (version stamped from the tag via `uv version`), creates the matching GitHub Release (body from `planning/releases/<tag>.md` when present, else GitHub's generated notes), and floats the `v0` action tag — in one job, PyPI first so a failed publish creates no Release. Replaces the old `release: published` `publish.yml` and the separate `tag-major.yml` (its `v0` logic is folded in as a `Float major tag` step, skipped on pre-releases). ([PR #35](https://github.com/modern-python/semvertag/pull/35))
- **The self-dogfooding auto-tagger runs in dry-run** (`.github/workflows/semvertag.yml`). It still exercises `action.yml` on every push to `main` but no longer pushes tags, so the only tags reaching the repo are deliberate, maintainer-pushed release tags — a hand-pushed tag is the sole release entry point.
- **The release runbook moved out of the docs site** into `CLAUDE.md` (maintainer-only); the stale `docs/contributing/release.md` (it still described an OIDC/tag-guard pipeline that no longer existed) was removed.

## Why a patch with no code change

The 0.8.x package code is unchanged since 0.8.0; nothing in `semvertag/` moved. This version exists to exercise and ship the new release machinery end-to-end against PyPI and the GitHub Release API. It was preceded by a throwaway pre-release (`0.8.1a1`) that validated the prerelease path (PyPI prerelease upload, generated-notes fallback, `v0` float correctly skipped).

## Release procedure (new, tag-driven)

Write these notes at `planning/releases/0.8.1.md`, then off green `main`:

```sh
git tag 0.8.1
git push origin 0.8.1
```

`release.yml` runs `just publish` (`uv version 0.8.1`, build, upload) to PyPI, then creates the GitHub Release from this file, then force-updates `v0` to the 0.8.1 commit. PyPI is irreversible and runs first; there is no CI gate (the tag is the commitment point). Pre-releases use the PEP 440 form (`0.8.1a1`).
